// 02 — Como Funciona

Dois Níveis de Explicação

Imagine uma linha de montagem da Ford — mas para código.

Quando um desenvolvedor termina de escrever código e faz o "push" para o GitLab (como salvar o trabalho na nuvem), normalmente um colega sênior precisa parar o que está fazendo, ler tudo e dar feedback. Isso pode levar horas — ou dias.

Com a Asystenc.ia, no exato momento do push, o GitLab envia um aviso automático para a nossa "empresa virtual". Em menos de 8 segundos:

O resultado é um Merge Request automático com o código melhorado, bugs corrigidos e vulnerabilidades removidas — como se um time de 5 engenheiros seniores tivesse revisado em paralelo.

Orquestração paralela de LLMs via webhooks n8n + GitLab Events API
// Fluxo de execução — n8n workflow 1. GitLab Webhook → POST /webhook/asystencia payload: { project_id, diff, commit_sha, branch } 2. Groq Node (Tech Lead) model: "llama3-70b-8192" latency: < 300ms output: { change_type, risk_level, routing_flags } 3. Parallel Branch Node (NVIDIA NIM — 3 workers simultâneos) ├─ Worker A: nvidia/llama-3.1-nemotron-70b → SecAudit ├─ Worker B: nvidia/mixtral-8x22b → PerfReport └─ Worker C: nvidia/llama-3.1-70b → QualityReport // 3 HTTP requests em paralelo → sem bloqueio de I/O 4. Merge Results Node (aguarda todos os workers) // Agrega os 3 JSONs em um único contexto 5. Gemini Node (Engineering Manager) model: "gemini-1.5-pro" context_window: 1.000.000 tokens // Recebe: diff original + 3 relatórios + histórico Supabase output: { patched_code, review_comment, confidence_score } 6. GitLab API Node POST /projects/:id/merge_requests // Abre MR com código corrigido + comentário detalhado 7. Supabase Node INSERT INTO reviews (sha, reports, decision, timestamp)
Por que Groq primeiro? O primeiro estágio precisa de latência mínima para classificar o payload e não desperdiçar contexto nos modelos NVIDIA com código irrelevante. A separação de responsabilidades entre velocidade (Groq) e profundidade (NVIDIA + Gemini) é a chave da arquitetura.
